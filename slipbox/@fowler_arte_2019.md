## 1. Refactoring: un primo esempio
- refactor first, add functionality then
- refactoring is useful when someone needs to understand the code and work on it after it has been written
- write tests first, refactor then
- make implicit knowledge explicit in the code itself[^explicit-logic]
- piccoli cambiamenti, test dopo ogni cambiamento rendono più facile il processo di debugging quando serve
- abbinare il ciclo di feedback loops molto brevi e ravvicinati all'uso di un sistema di versionamento. Fowler parla di piccolo commit in locale e poi combinazioni di commit più consistenti da mandare alla repo condivisa. Come si combinano i commit?
- automatizzare il refactoring "Extract function"!
- use variable names that contain information about the type of the variable, especially for function parameters (eg "aPerformance")
- Replace Temp With Query: this refactoring is used to remove local variables from a function when they can be computed using variables that are already inside the scope rather than passed. We replace the temporary variable with a function call that simply extracts the value and returns it.
- Inline Variable: this is another refactoring that simplifies the local scope of functions. We replace any temporary variable, that is generated by a function and never modified, with an inline function call.
- Split loop is a refactoring that intuitively seems not a good idea because of efficiency, but can be very useful to apply Extract Function.
- Change Function Declaration and Slide Statements are changes that are so small that I wouldn't have called them refactoring steps at all; still, Fowler does (and even runs tests and commits after doing them!)
	- when a test fails, he simply goes back to the last working commits and does things again but in smaller, simpler steps (the right way to use a version control software!)
- Fowler writes every function in such a way that it always returns one output and has no side effects. He explicitly says that he prefers not to modify existing state (=input data) even if it would be possible to do so. This is also the logic that is consistently applied in Python and particularly in Pandas[^pandas].
- Fowler separates the logic of data building from that of data presentation (rendering). This allows to keep the rendering part as it is and to modify the data building part provided that the format of the data passed is the same. Importantly, we can add a test on the data format to have a stronger codebase.
- Replace Conditional with Polymorphism: this refactoring exploits the possibility to define hierarchies of types given by OOP to introduce more flexibility in the computation of attributes. 
- Common phases of the refactoring process: reading the code and having ideas about how it works. Transfer those ideas from our head to the code by imposing structure. Apply deeper refactoring steps taking advantage of the understanding and structuring done before.
- Most important thing to learn: the refactoring loop is really small and quick. It's important to have tests implemented and to compile-test-commit frequently. This allows to leave the codebase always in a working state - and to stop and resume the work whenever we want!
