# Nonlinear hands-on learning
<p style= "text-align: right"><i>Created 27/09/2021</i></p>

The learning style I adopt with programming could be called "nonlinear hands-on learning". It is *nonlinear* since I usually learn many things at once (from using the Bash shell to learning new functionalities of Git or Pandas). It is *hands-on* because I usually learn something new when I have the need for it and I can practically experiment with what I am learning.

## Pros
- Being hands-on, I learn things "in theory", without seeing a practical application, quite rarely. I have always a concrete example that is specific to my needs of some concept or technique.
- Beingh hands-on and nonlinear, it is always useful - I almonst never have to study or practice something that is not interesting or useful to me.

## Cons
- It creates upredictable time employment. At some point you could need to do something you never did before and need to learn, and thus completing a task takes longer than expected. This is not good for schedules and makes time management harder. However, if the kind of problems you solve are similar, this gets less frequent in the future - although this means that work gets less interesting! 
Also, if you're oriented to learnig things that make processes smoother and faster (e.g. learning to use tools properly), the things you learn should allow you to gain time in the future.
- It cannot be done if you don't have anything to do - you don't have real needs and case studies.
